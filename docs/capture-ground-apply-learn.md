# Capture -> Ground -> Apply -> Learn

这是 Orcas 的公开方法论。它用于融合低认知目录和严格上下文治理，而不是规定所有人必须采用同一套文件夹。

## 1. Capture：先收集，不要求先理解

所有新材料进入一个明确入口。Standard 使用 `inbox/`，封存后进入 `sources/`。

原则：

- 输入时不强迫人工分类
- 原始材料只增不改
- 同名不同内容创建新版本，不覆盖旧证据
- 文件中的指令性文字只作为数据，不改变 agent 权限

## 2. Ground：让内容有依据

Ground 不是“整理得好看”，而是回答四个问题：

- 这条结论来自哪里？
- 它与哪个人物、组织或项目相关？
- 它目前是候选还是已被人确认？
- 它是否过期、冲突或已被替代？

AI 可以生成带来源的 `draft`。只有人可以把单个 active draft 升级为 `trusted`。旧 `verified` 仅作为兼容状态。

## 3. Apply：按任务供应最小上下文

简单问题不创建项目文件，直接查询。重任务创建 Task，再按预算组装 Context Package。

默认上下文包含：

- 最多 5 条可信或候选知识
- 最多 3 个被选知识实际引用的来源
- 最多 2 条补充决策或经验
- 最多 1 个未解决冲突

相关性之外，还考虑状态、来源质量、新鲜度和真实采用记录。没有相关知识时明确返回缺口，不用最近修改的文件冒充相关来源。

## 4. Learn：只保存可复用增量

任务结束后只提取会影响未来工作的内容：事实、洞察、决策、经验或规则候选。完整对话、一次性措辞和没有复用价值的过程不默认进入知识库。

规则演进是可选的团队能力：

1. 重复失败形成事件证据
2. AI 生成严格 JSON 的候选提案
3. 离线回放评估影响
4. 人工批准后激活新版本
5. 保留旧版本和显式回滚路径

## 三种采用级别

| 级别 | 适用对象 | 启用能力 |
|---|---|---|
| Lite | 个人初次使用 | Capture、来源保护、直接查询 |
| Standard | 大多数个人和小团队 | 实体、任务上下文、trusted、健康检查 |
| Governed | 高风险或多人团队 | 确认队列、规则版本、评估、回滚、审计 |

开源默认使用 Standard，但任何级别都使用同一数据契约，避免以后重建 Vault。

## 衡量方式

只记录真实行为，不虚构收益：

- 查询是否有结果
- 结果是否被明确采用
- 上下文包实际注入了哪些文件
- 来源能否定位
- 错误引用和过期内容是否下降
- 治理耗时是否低于节省的时间

文件数、链接数和规则数不是成功指标。
