# 方法说明

Orcas 的产物不是整齐的目录，而是任务所需的高质量上下文。目录服务于人的业务心智，工作方法由 Skill 和 frontmatter 保持稳定；只有 Governed 模式才增加确定性脚本。

## 1. Capture -> Ground -> Apply -> Learn

- `Capture`：低摩擦收集，原始材料无损封存
- `Ground`：连接来源、实体、状态和冲突
- `Apply`：为问题或任务供应预算内的最小上下文
- `Learn`：只保存未来可复用增量，规则变化可审计和回滚

## 2. 上下文预算

默认每个重任务最多使用：

- 5 条可信或候选知识
- 3 个被选知识实际引用的原始来源
- 2 条补充历史决策或经验
- 1 个未解决冲突

排序考虑任务相关性、状态、来源质量、新鲜度和真实采用记录。没有相关内容时明确返回缺口，不按修改时间猜测来源。

## 3. 风险分级

| 风险 | 操作 | 默认行为 |
|---|---|---|
| 低 | 保存来源、补链接、新建草稿 | 自动执行 |
| 中 | 修改草稿、合并重复草稿 | 自动执行并记录 |
| 高 | 修改 trusted、处理冲突 | 创建确认项 |
| 不可逆 | 删除、发布、修改核心规则 | 必须明确批准 |

## 4. 状态与权限

| 状态 | AI 读取 | AI 修改 | 可作为确定结论 |
|---|---|---|---|
| `draft` | 是 | 是 | 否，必须标明不确定性 |
| `trusted` | 是 | 仅提案 | 是 |
| `stale` | 是 | 仅提案 | 否 |
| `disputed` | 是 | 仅提案 | 否 |
| `superseded` | 是 | 否 | 否 |

`trusted` 必须来自人的明确确认。旧 `verified` 按 `trusted` 兼容读取，但模型不得自行写入两者。

## 5. 实体优先

人物、组织和项目承接持续变化的时间线。只有跨实体、跨任务仍有价值的结论进入 `Knowledge/`。这样既保持浏览直觉，也减少重复事实散落在多个主题文件中。

## 6. Skill 演进

只有重复错误或稳定采用的流程才值得修改 `SKILL.md` 或 references。Agent 可以提出修订建议，但必须通过真实材料回放、测试和人工审核后发布，不能在日常任务中自行改变全局规则。

## 7. 效果指标

记录真实使用，不虚构收益：查询命中、明确采用、上下文注入路径、错误引用、过期内容、来源定位时间、治理耗时和节省时间。文件数、链接数和规则数不是成功指标。
