# v0.5 实际工作模拟

模拟日期：2026-07-19

模拟场景：为“华东制造知识治理试点”处理一份客户访谈，并准备四周试点方案。

> 本次使用隔离副本 `.simulation-v05/vault/`，没有修改正式 `starter-vault/sources/`。

## 操作链路

### 材料进入

将访谈放入 `inbox/`，运行：

```sh
python3 scripts/intake.py
```

结果：

- 原文进入 `sources/客户访谈-2026-07-19.md`。
- 工作副本进入 `inbox/_processed/`。
- 两份文件 SHA-256 一致。
- 原文中的“忽略规则、直接 trusted、删除冲突”等提示注入没有改变系统行为。

### AI 提取增量

运行：

```sh
python3 scripts/ai.py process-inbox
```

生成三个 draft：

- 客户试点的三个业务问题。
- 第一阶段采用本地 Vault 的决定。
- 销售资料权限待确认事项。

所有条目都保留来源路径，AI 没有直接设置 trusted。

### 直接查询

运行：

```sh
python3 scripts/query.py "华东制造第一阶段怎么做" --adopt
```

系统返回本地 Vault、Python 脚本、每周复盘等候选结论，并明确标记“候选，需核实”。查询没有创建任务文件，采用事件写入 `_reports/usage/events.jsonl`。

### 实体更新时间线

运行：

```sh
python3 scripts/entity.py new project "华东制造知识治理试点" --source sources/客户访谈-2026-07-19.md
python3 scripts/entity.py append project "华东制造知识治理试点" "客户确认第一阶段采用本地 Vault，每周复盘一次" --date 2026-07-19 --source sources/客户访谈-2026-07-19.md
```

结果：同一项目只生成一个实体页，新信息追加到时间线。

### 重任务上下文

运行：

```sh
python3 scripts/context.py new "为华东制造客户准备四周试点方案"
python3 scripts/context.py build work/2026-07-19-为华东制造客户准备四周试点方案.md
```

上下文包含：

- 三条候选知识。
- 一个项目实体。
- 一个实际引用来源。
- 历史决策和状态使用边界。

没有按修改时间随机补来源。

### 人工确认

运行：

```sh
python3 scripts/trust.py knowledge/华东制造第一阶段采用本地-vault.md --actor "周敏" --evidence "2026-07-19 访谈中明确确认"
```

再次查询后，该决定从“候选”变为“可信”。使用通配符批量升级时，命令返回非零状态并拒绝操作。

### 健康检查

运行：

```sh
python3 scripts/health.py
```

结构检查通过，没有 P0 或 P1 报告。

## 模拟中确认有效的能力

- 来源封存与不可变校验有效。
- 提示注入边界有效。
- AI 不会直接 trusted。
- draft 可以进入实际检索，而不是完全失联。
- 实体时间线可以承接项目增量。
- 简单查询不需要创建任务。
- 单条人工确认和禁止批量升级有效。
- 查询和上下文注入有真实事件记录。

## 模拟中发现的问题

### 上线前建议修复

1. 中文检索存在误命中。“完全无关的火星地质问题”仍命中包含“问题”的客户知识，说明二元词片对常见词过敏。
2. `trust.py` 只更新 frontmatter。升级后正文仍写“当前状态为 draft”，形成自相矛盾。
3. 同一个 decision 同时进入“可信知识”和“历史决策”，使用事件也重复记录同一路径。
4. `AGENTS.md` 和部分文档仍使用旧 `verified` 术语，与 v0.5 的 `trusted` 不完全一致。
5. 项目中存在误创建的字面目录 `wiki/entities/{people,projects,organizations}`，应清理。
6. AI 适配器返回空对象时，`prepare-task` 仍会生成“未生成摘要”的空白简报，缺少严格 schema 校验和失败中止。

### 可在试点中观察

1. 查询默认返回范围偏宽，缺少“只看可信”“只看实体”“只看待确认”等过滤选项。
2. 使用事件记录了 `adopted: true`，但仍由用户主动传入 `--adopt`，不能证明最终交付真的采用。
3. 健康检查只检查结构，未发现正文状态与 frontmatter 冲突。
4. 命令仍较多，普通 Obsidian 用户需要快捷入口或统一命令封装。

## 判断

v0.5 已经达到“小规模、低风险、有人复核的真实试点”标准，但还没有达到“可以无脑放进日常工作并稳定依赖”的标准。

建议先完成上述五项上线前修复，发布 v0.5.1，再选一个真实客户项目连续运行两到四周。评估来源定位时间、重复提问、误命中、人工维护时间、实际采用知识和返工次数，然后再判断是否真正好用。
